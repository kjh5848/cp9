# 재발방지 와이 스킬 (Why Analyzer)

동일한 에러나 요구사항 누락 등 반복적인 문제가 발생하는 것을 방지하기 위해 근본 원인을 찾아내고 구조적인 개선 방안을 도출하는 스킬입니다.

## 🚀 실행 조건 (Trigger)

사용자가 다음과 같은 질문이나 명령을 입력하면 **즉시** 이 스킬을 실행합니다.
- "와이"
- "왜?"
- "왜그런거야?"

## 🛠️ 실행 워크플로우 (Execution Steps)

스킬이 트리거되면 다음 파이프라인에 따라 작업을 수행하십시오.

### 1. 다각도 원인 분석 (Root Cause Analysis)
단순한 코드 오타나 표면적 버그 수정을 넘어서, 근본적인 원인을 다음 방면에서 종합적으로 조사합니다.
- **Rules (규칙)**: 현재 프로젝트에 설정된 글로벌 룰이나 프롬프트 지침과 상충하는 부분은 없는가?
- **Workflows (워크플로우)**: 에이전트 작업 순서나 기존 워크플로우에 논리적 결함은 없는가?
- **Architecture (아키텍처 구조)**: 시스템 설계, 폴더 구조 (예: FSD), 컴포넌트 간 의존성에 문제가 있는가?
- **Code Quality (코드 품질)**: 안티 패턴, 하드코딩, 예외 처리 누락 등 코드 베이스 자체의 결함이 있는가?
- **기타**: 인프라, 환경 변수 등 외부 요인

### 2. 문제 코드 제거 및 개선 방안 도출
- 분석 결과를 바탕으로 실제 문제가 되는 코드, 로직, 또는 텍스트 부분을 식별하여 삭제 및 수정 방안을 마련합니다.
- 문제 해결을 넘어, 이후 동일한 문제가 발생하지 않도록 **재발 방지를 위한 구조적 개선 방안 및 패턴** 을 사용자에게 제시합니다.

### 3. 와이 로그 파일 (Why Log File) 작성
- 파악한 원인과 문제가 되었던 코드를 문서화하여 별도의 로그 파일로 저장해야 합니다.
- **저장 위치**: 프로젝트 루트 경로 아래 `.agents/why-logs/` 디렉토리에 저장합니다. (필요 시 디렉토리를 생성합니다)
- **파일명 규칙**: `YYYY-MM-DD_[문제의_핵심_요약].md` (예: `2026-03-18_api_timeout_issue_cause.md`)
- **로그 작성 템플릿**:
  ```markdown
  # [문제 요약 제목]

  ## 🚨 발생 문제
  - (문제가 발생한 구체적인 현상과 내용)

  ## 🔍 원인 분석
  - **분석 관점**: (Architecture, Code Quality, Rules 등)
  - **근본 원인**: (왜 이런 문제가 나타났는지 상세 설명)

  ## 💡 해결 및 재발 방지 방안
  - (문제가 된 로직의 수정/제거 내용)
  - (이후 동일 상황 발생 시 따라야 할 Best Practice 및 개선 전략)
  ```

### 4. 경로 참조 응답 (Response Formatting)
- 와이 로그 파일을 생성한 후, 사용자에게 응답할 때는 **로그 파일의 내용을 길게 다시 나열하지 않습니다.**
- 오직 생성된 **와이 로그 파일의 절대 경로를 제시**하여, "원인 분석 및 개선 방안을 다음 파일에 기록했습니다"라고 안내합니다.
- **목적**: 이후 비슷한 상황이 발생했을 때 에이전트가 이 디렉토리 경로의 파일들을 먼저 조회(`view_file` 등 활용)하여 과거의 트러블슈팅 이력을 문맥으로 활용하도록 하기 위함입니다.
- **모든 응답 및 작성 문서는 반드시 한국어로 작성** 합니다.

## ⚠️ 글로벌 주의 사항 (Constraints)
프로젝트 글로벌 룰에 따라 다음 사항을 반드시 지켜야 합니다.
- 모든 문서 작업과 대화 출력은 한국어로 번역해서 수행합니다.
- 마크다운 볼드체(`**`)를 사용하여 텍스트(괄호) `텍스트() `, 따옴표 `"" ` 문구를 강조할 때는 문자 뒤에 반드시 공백을 넣어야 합니다. (예: `**"강조"** 함`, 오류 예: `**"강조"**함`)
- 모든 주석은 한국어로 기입합니다.
